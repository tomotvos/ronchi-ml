# Ronchi-ML: Analysis of Current Approach and Recommended Next Steps

**Revision:** 2  
**Date:** 2026-05-03  
**Repository:** https://github.com/tomotvos/ronchi-ml  
**Scope:** Evaluate whether the current synthetic-training/CNN approach for Ronchi image analysis is technically sound, whether it can be extended to imperfect mirrors, and what should be done next before adding defects. This revision explicitly incorporates the fact that held-out synthetic images also show imperfect inference, even when the real-to-synthetic image gap is removed.

---

## 1. Executive conclusion

The current approach is still directionally sound, but the new information changes the priority order.

The earlier assessment emphasized the synthetic-to-real gap as the dominant known risk. That remains true for the eventual product, because real Ronchi photographs differ strongly from ideal synthetic renderings in illumination, blur, contrast, line width, pupil crop, rotation, and colour response. However, if held-out synthetic images are also not matching or predicting as closely as expected, then the immediate blocker is not only domain gap. The project first needs to prove that the model and matching pipeline can solve the **clean synthetic inverse problem**.

The revised diagnosis is:

> Before adding real-image robustness or mirror defects, establish a synthetic upper-bound baseline. If the model cannot predict correction and offset accurately on withheld synthetic images generated by the same simulator, adding realistic preprocessing or defects will only hide the underlying problem.

This does not mean the CNN approach is wrong. It means the next phase should be more diagnostic and experimental:

1. audit the manifest, labels, generated filenames, preprocessing, and train/validation split logic;
2. verify that the model can overfit a tiny synthetic subset;
3. test clean synthetic generalization under several structured holdout regimes;
4. isolate whether the difficulty is correction, offset, `f/#`, line count, or their interaction;
5. compare direct regression, raw nearest-neighbour, learned-embedding nearest-neighbour, and oracle-offset baselines;
6. only then proceed to synthetic-realistic augmentation, real-image normalization, and defects.

The most important practical change is this:

> Clean synthetic holdout performance should become the first gate. The project should not introduce defects until clean no-defect synthetic inference is predictably good.

## 2. What the current implementation is doing

The repository currently describes Ronchi-ML as a CNN-based estimator for Ronchi test analysis that predicts:

- `p_corr`: parabolic correction;
- Ronchi offset;
- conditioned on mirror/grating geometry, specifically mirror `f/#` and `lines` across the mirror diameter.

The manifest schema stores:

```json
{
  "image": "output/images/ronchi_2.5_-0.050_0.4_680.png",
  "meta": {
    "f": 2.5,
    "offset": -0.05,
    "lines": 680.0
  },
  "labels": {
    "p_corr": 0.4
  }
}
```

The documented architecture is a two-head CNN:

- a single-channel image backbone;
- conditioning projection for `[f_norm, lines_norm]`;
- fusion into a 256-dimensional representation;
- one head for `p_corr`;
- one auxiliary head for normalized offset.

So the system is not currently a pure lookup table. It is a supervised regression model with an auxiliary offset task. Internally, it already creates a learned vector representation. That representation is not yet exposed as a retrieval embedding, but conceptually the model is already “vectorizing” the image.

## 3. Does imperfect held-out synthetic performance change the assessment?

Yes. It changes the triage, not the whole strategy.

The prior assessment was:

> The architecture is plausible; the main problem is likely the synthetic-to-real gap.

The revised assessment is:

> The architecture is plausible, but the first unresolved problem is whether the current training/matching pipeline learns the clean synthetic optical mapping at all. The synthetic-to-real gap remains important, but it is no longer safe to treat it as the only or primary explanation.

This matters because clean synthetic data removes many photographic nuisance variables:

- no uneven illumination;
- no camera blur;
- no colour cast;
- no pupil-detection uncertainty;
- no thresholding ambiguity;
- no rotation/crop artifacts unless deliberately generated.

If inference is still surprisingly wrong on held-out synthetic images, likely causes include:

1. label or manifest mismatch;
2. train/inference preprocessing mismatch;
3. offset/correction ambiguity;
4. insufficient model capacity or overly aggressive downsampling;
5. multi-task conflict between `p_corr` and offset;
6. weak generalization across synthetic grid points;
7. data split that tests extrapolation rather than interpolation;
8. bugs in normalization or denormalization;
9. sign convention mismatch for offset or inside/outside radius of curvature;
10. an inverse problem that is genuinely ambiguous from a single Ronchi image.

The highest-value work now is to determine which of those is actually happening.

## 4. The correct mental model: synthetic upper bound first

The project should explicitly measure four data domains:

```text
A. synthetic-clean random holdout
B. synthetic-clean structured/grid holdout
C. synthetic-realistic holdout
D. real-curated images
```

Each domain answers a different question.

### 4.1 Synthetic-clean random holdout

This is the easiest test. Images are generated by the same simulator and drawn from the same distribution as training.

If performance is poor here, suspect:

- broken labels;
- broken image loading;
- train/inference mismatch;
- model undertraining;
- optimizer or normalization problem;
- model capacity problem;
- target ambiguity even within the synthetic generator.

### 4.2 Synthetic-clean structured/grid holdout

This holds out specific values or ranges of `p_corr`, offset, `f/#`, or line count.

If random holdout is good but structured holdout is poor, suspect:

- the model is interpolating/memorizing nearby grid points;
- the synthetic grid is too sparse or uneven;
- the model has not learned the underlying optical relationship;
- conditioning variables are not being used effectively;
- `p_corr` and offset effects are entangled.

### 4.3 Synthetic-realistic holdout

This applies real-photo-like corruptions to synthetic images.

If clean synthetic is good but synthetic-realistic is poor, then the previous domain-gap diagnosis is confirmed.

### 4.4 Real-curated images

This is the eventual smoke test. It should not be the primary diagnostic until clean synthetic performance is understood.

## 5. Why synthetic holdout can still be difficult

It is tempting to expect synthetic holdout accuracy to be almost perfect. It should be much better than real-image accuracy, but there are reasons it may still fail.

### 5.1 Correction and offset may be partially degenerate

Ronchi band shape changes with both correction and tester offset. If offset is not known and must be inferred from the image, the model may trade correction error against offset error.

For example:

```text
slightly different p_corr + slightly different offset
```

may produce visually similar bands, especially at some `f/#` and line-count regimes.

That does not invalidate the project, but it means the output should eventually be probabilistic or top-k, not a single overconfident value.

### 5.2 Dense grid neighbours can be visually similar

If the synthetic generator samples many nearby `p_corr` and offset values, adjacent examples may differ subtly. A small image-processing or model error can jump the prediction by several grid steps.

This is especially likely if:

- the rendered bands are thick;
- the image is resized to 320 × 320;
- curvature differences are subtle;
- the model uses global pooling that discards some spatial detail;
- the loss treats all scalar errors uniformly but the image evidence is locally weak.

### 5.3 The model may be learning rendering artifacts rather than optical state

Even within synthetic data, the model can latch onto artifacts of the generator, filename ordering, anti-aliasing, band phase, or image scaling rather than robust fringe geometry.

Structured holdouts reveal this better than random splits.

### 5.4 Multi-task training can help or hurt

The current loss is:

```text
p_corr_loss + off_weight × offset_loss
```

Auxiliary offset prediction is physically sensible. But if offset prediction is harder than correction prediction, the shared backbone may be pulled toward features that hurt `p_corr` accuracy.

This should be tested directly with a `p_corr`-only baseline.

### 5.5 The current representation may be too compressed

The documented architecture uses several stride-2 convolutions and then adaptive pooling to 4 × 4. That is compact and efficient, but Ronchi interpretation depends on geometry of curved bands across the pupil. If subtle curvature differences matter, the model may benefit from preserving more spatial information.

Potential architecture experiments:

- increase input size from 320 to 512 or 640;
- use adaptive pooling to 8 × 8 instead of 4 × 4;
- add coordinate channels or radial coordinate maps;
- use a lightweight ResNet backbone;
- use a Siamese/contrastive embedding objective for retrieval;
- use a shape-aware preprocessing channel such as signed distance transform.

Do not do all of these at once. First establish whether the current model fails even on a tiny overfit test.

## 6. Metadata does affect the architecture

The intuition “just find the closest image and output the metadata” is useful at the product level, but metadata affects training and matching in several ways.

### 6.1 Metadata defines what “same” means

If two images have similar pixels but different optical states, the labels tell the model they should not be considered equivalent. Without labels or metric learning, a nearest-neighbour system does not know which visual differences matter.

For example:

```text
same p_corr, different offset
```

and

```text
different p_corr, same offset
```

may both change band shape. The metadata is what tells the model which axis of variation corresponds to correction and which corresponds to offset.

### 6.2 Conditioning metadata changes the inverse problem

The current model conditions on `f` and `lines`, but not on offset because offset is predicted. That means the model is solving:

```text
image + f + lines -> p_corr + offset
```

An alternative diagnostic model is:

```text
image + f + lines + known offset -> p_corr
```

This “oracle offset” baseline is extremely useful. If it performs much better, then offset ambiguity is a major source of error.

### 6.3 Retrieval must compare like with like

If a retrieval index contains synthetic images across many `f/#`, line counts, offsets, and correction levels, a naive nearest-neighbour search may retrieve examples that are visually similar for the wrong reason.

Better retrieval options:

1. filter the synthetic catalog by user-provided geometry before searching;
2. include geometry in the learned embedding;
3. compare only within plausible offset bands;
4. report top-k clusters, not only top-1.

## 7. Immediate diagnostic experiments

These experiments should be done before adding defects.

### Experiment 1: manifest and label audit

Create a script:

```text
src/audit_manifest.py
```

It should check:

- every image path exists;
- every image can be loaded;
- every image is the expected size or resizeable;
- parsed filename parameters match JSON metadata where possible;
- `p_corr` values are in the expected range;
- offset sign convention is consistent;
- `f` and `lines` distributions are sensible;
- no duplicate image paths with different labels;
- no duplicate labels pointing to different images unless intentional;
- train/validation/test splits do not overlap by path;
- train/validation/test splits do not unintentionally share near-identical generated states.

Output:

```text
runs/audit/<timestamp>/manifest_audit.json
runs/audit/<timestamp>/summary.md
runs/audit/<timestamp>/sample_contact_sheet.png
```

Acceptance:

- zero missing images;
- zero duplicate path/label conflicts;
- metadata/filename agreement, if filenames encode parameters;
- clear histograms for `p_corr`, offset, `f`, and `lines`.

### Experiment 2: tiny overfit test

Train the current model on 16, 32, 64, and 128 synthetic-clean examples. Evaluate on the same examples.

Expected result:

```text
training MAE should approach near-zero
```

If it cannot overfit a tiny clean dataset, likely causes are:

- preprocessing mismatch;
- target scaling issue;
- optimizer/loss problem;
- architecture bug;
- labels do not correspond to images.

This is the most important sanity test.

### Experiment 3: p-only baseline

Train a model with only the `p_corr` head and no offset loss.

Compare:

```text
current multitask model: p_corr + offset
p-only model: p_corr only
```

If p-only is much better, the auxiliary offset task is hurting correction estimation.

If multitask is better, keep the auxiliary head.

### Experiment 4: oracle-offset baseline

Train/evaluate a variant where offset is provided as conditioning input, not predicted.

Model input:

```text
image + f + lines + offset -> p_corr
```

This answers a critical question:

> Is the model bad at reading correction, or is it bad at jointly disentangling correction and offset?

If oracle offset dramatically improves accuracy, then the final CLI should strongly encourage users to provide an offset estimate or should perform an explicit offset scan.

### Experiment 5: single-axis sweeps

Generate controlled sweeps:

```text
fixed f, fixed lines, fixed offset, vary p_corr
fixed f, fixed lines, fixed p_corr, vary offset
fixed p_corr, fixed offset, fixed lines, vary f
fixed p_corr, fixed offset, fixed f, vary lines
```

Plot predicted vs true values.

Expected behaviour:

- `p_corr` predictions should be monotonic when only `p_corr` changes;
- offset predictions should be monotonic when only offset changes;
- predictions should not jump discontinuously between neighbouring grid points;
- residuals should not cluster around particular offsets or line counts.

### Experiment 6: raw nearest-neighbour baseline

Implement a simple baseline:

- preprocess all synthetic images identically;
- compare query to catalog using pixel MSE or normalized cross-correlation;
- optionally allow small vertical rolls and rotations;
- return top-k metadata.

This is not the recommended final method, but it is a valuable diagnostic.

Interpretation:

- If raw matching beats the CNN on synthetic-clean holdout, the CNN training pipeline is suspect.
- If CNN beats raw matching on synthetic-clean but fails on synthetic-realistic, domain randomization is needed.
- If both fail on structured holdouts, the inverse problem or synthetic grid may be underconstrained.

### Experiment 7: embedding retrieval from the current model

Expose the 256-dimensional fused representation and build a nearest-neighbour index over synthetic images.

Compare:

```text
head regression prediction
embedding top-1 neighbour
embedding top-5 neighbour consensus
raw pixel top-5 neighbour consensus
```

This gives immediate insight into whether the learned representation is physically meaningful.

## 8. Revised evaluation plan

The existing evaluation target of `RMSE <= 0.05` on validation is a reasonable early POC criterion, but it is not enough.

Use a matrix of evaluations:

| Split | Purpose | Expected result |
|---|---|---|
| Tiny train-overfit | Code/model sanity | Near-zero train error |
| Synthetic-clean random | Same-distribution generalization | Very low error |
| Synthetic-clean grid holdout | Interpolation/generalization | Low to moderate error |
| Synthetic-clean regime holdout | Hard generalization | Explainable degradation |
| Synthetic-realistic | Robustness to photographic nuisance | Moderate degradation |
| Real-curated | Product smoke test | Plausible/stable outputs |

Recommended metrics:

```text
p_corr_MAE
p_corr_RMSE
offset_dimless_MAE
offset_sign_accuracy
p_corr_error_by_offset_bin
p_corr_error_by_f_bin
p_corr_error_by_lines_bin
prediction_variance_under_TTA
top1_match_within_tolerance
top5_match_within_tolerance
regression_vs_retrieval_disagreement
```

Suggested initial gates:

```text
Tiny overfit:                 p_corr_MAE near 0
Synthetic-clean random:       p_corr_MAE <= 0.02 to 0.03
Synthetic-clean grid holdout: p_corr_MAE <= 0.03 to 0.05
Synthetic-realistic holdout:  p_corr_MAE <= 0.05 to 0.08 initially
```

These thresholds are provisional. The important point is not the exact number; it is that clean synthetic performance should establish a strong upper bound.

## 9. Data splitting recommendations

Random splits are useful but insufficient.

### 9.1 Random image split

Fast iteration only. This is the easiest split and can be overly optimistic if the synthetic grid is dense.

### 9.2 State-based split

Split by optical state, not image path.

Define state as approximately:

```text
(f, lines, offset, p_corr, defect parameters)
```

This prevents near-duplicate states from leaking across train and validation.

### 9.3 Axis holdout

Hold out entire values or ranges of:

- `p_corr`;
- offset;
- `f/#`;
- line count.

### 9.4 Regime holdout

Hold out combinations such as:

```text
low f/# + high line count + positive offset
high f/# + low line count + negative offset
near-parabola + small offset
strong undercorrection + large offset
```

### 9.5 Corruption holdout

Once synthetic-realistic images are introduced, hold out corruption families:

- train on blur/noise/brightness;
- test on rotation/vignetting/crop jitter;
- reverse and compare.

## 10. Architecture assessment

The current CNN is a reasonable first architecture. It should not be replaced until it has been tested properly.

However, if clean synthetic holdout remains poor after audit and training fixes, consider these architecture changes in this order.

### 10.1 Increase spatial preservation

Ronchi interpretation is geometric. Try:

- input size 512 instead of 320;
- adaptive pooling 8 × 8 instead of 4 × 4;
- less aggressive stride early in the network;
- modest ResNet-style backbone.

### 10.2 Add coordinate information

CNNs are translation-equivariant, but Ronchi interpretation depends on radial position within the pupil.

Possible additional input channels:

```text
x coordinate normalized to [-1, 1]
y coordinate normalized to [-1, 1]
radius from pupil centre
pupil mask
```

This may help the model distinguish a central curvature feature from an edge feature.

### 10.3 Use signed-distance or soft-mask inputs

A hard binary image may discard useful subpixel geometry. A signed distance transform can make line geometry smoother and more learnable.

Candidate one-channel inputs:

- binary mask;
- signed distance transform;
- blurred binary mask;
- skeleton distance map.

Candidate multi-channel input:

```text
channel 1: normalized grayscale or binary mask
channel 2: signed distance transform
channel 3: pupil/radial coordinate map
```

### 10.4 Use retrieval as a diagnostic before making it the product

Expose embeddings now, but first use them to debug.

Questions to answer:

- Do neighbouring embeddings correspond to neighbouring `p_corr` values?
- Do embeddings cluster by `f/#` or line count instead of correction?
- Are offset differences dominating the embedding?
- Do top-k retrieved images visually resemble the query?

## 11. The role of vector search

A vector database in the Open Brain sense is not the missing core mechanism. The current CNN already creates a learned vector internally.

The missing piece is exposing and evaluating that vector.

Recommended approach:

```text
1. train supervised CNN on synthetic images;
2. expose fused 256-dimensional embedding;
3. build a synthetic index from embeddings;
4. retrieve top-k synthetic neighbours;
5. compare neighbour metadata to direct regression output;
6. later add contrastive training so embedding distances encode optical similarity more explicitly.
```

This is better than generic image embeddings because the representation is trained on Ronchi-specific labels and metadata.

### 11.1 Retrieval should be filtered or conditioned

For CLI inference, the user will provide:

- image;
- mirror diameter;
- approximate radius of curvature or focal ratio;
- grating LPI;
- possibly approximate tester offset.

Use that information to restrict retrieval:

```text
same or nearby f/#
same or nearby lines across diameter
plausible offset range if an offset hint is available
```

Then perform nearest-neighbour search inside that subset.

### 11.2 Return top-k, not just top-1

A single best match can be misleading. Return multiple candidates and a consensus.

Example:

```json
{
  "prediction": {
    "p_corr": 0.84,
    "offset_dimless": 0.012
  },
  "retrieval_consensus": {
    "median_p_corr_top5": 0.85,
    "median_offset_dimless_top5": 0.011,
    "spread_p_corr_top5": 0.04
  },
  "top_matches": [
    {
      "rank": 1,
      "similarity": 0.94,
      "p_corr": 0.83,
      "offset_dimless": 0.012,
      "image": "output/images/...png"
    }
  ],
  "warnings": []
}
```

If regression and top-k disagree, the CLI should say so.

## 12. Preprocessing recommendations remain important, but later

The attached real images still make it clear that real-image normalization will be important. The existing preprocessing tools are directionally appropriate:

- grayscale/Otsu preprocessing;
- signed-distance-map resizing;
- fixed-range thresholding;
- full real-frame pipeline with pupil crop, illumination correction, denoising, adaptive thresholding, and optional signed-distance resizing.

But given imperfect synthetic holdout performance, preprocessing should not be the first place to spend effort.

Recommended order:

```text
1. clean synthetic diagnosis
2. synthetic holdout generalization
3. embedding/retrieval diagnostics
4. synthetic-realistic domain randomization
5. real-image preprocessing bake-off
6. defect introduction
```

The “normalize to thin lines” idea should be treated as an experiment, not an assumption. Centerline shape is important, but skeletonization can discard useful information such as spacing, phase, line width, and edge behaviour.

---

## 13. Why perfect mirrors first remains correct

The new information strengthens, rather than weakens, the case for staying with no-defect mirrors for now.

If the system is not yet reliable for defect-free synthetic holdouts, adding defects will create confounded errors:

- a turned edge may be misread as a correction/offset shift;
- a low zone may be absorbed into global `p_corr` error;
- preprocessing artifacts may look like defects;
- the model may learn simulator-specific defect artifacts instead of optical signatures.

The first defect family should be introduced only after the no-defect synthetic pipeline passes the clean holdout gates.

## 14. Revised implementation roadmap

### Milestone A: audit and reproducibility

Goal: make the current dataset and training run inspectable.

Tasks:

1. Add `audit_manifest.py`.
2. Add deterministic split files saved to disk.
3. Save training config, manifest hash, git commit, and metrics per run.
4. Generate batch contact sheets from training and validation images.
5. Confirm image polarity, resize, normalization, and label ranges.

Gate:

```text
No unresolved manifest, label, split, or image-loading issues.
```

### Milestone B: tiny overfit sanity

Goal: prove the model can memorize simple synthetic examples.

Tasks:

1. Train on 16, 32, 64, 128 examples.
2. Evaluate on the same examples.
3. Plot training predictions vs labels.
4. Repeat with p-only and multitask versions.

Gate:

```text
Training p_corr_MAE should become near-zero on tiny clean subsets.
```

If this fails, stop and debug. Do not add domain randomization or defects.

### Milestone C: clean synthetic generalization

Goal: establish the true synthetic upper bound.

Tasks:

1. Random synthetic split.
2. State-based synthetic split.
3. Axis holdout split.
4. Regime holdout split.
5. Residual plots by `p_corr`, offset, `f`, and `lines`.

Gate:

```text
Synthetic-clean random p_corr_MAE <= 0.02 to 0.03
Synthetic-clean grid/state holdout p_corr_MAE <= 0.03 to 0.05
```

### Milestone D: ambiguity and offset diagnosis

Goal: identify whether offset is the main source of error.

Tasks:

1. Train p-only model.
2. Train oracle-offset-conditioned model.
3. Compare with current multitask model.
4. Implement explicit offset scan baseline.
5. Evaluate prediction stability when offset is fixed vs unknown.

Gate:

```text
Known-offset or scanned-offset inference should materially reduce ambiguity, or demonstrate that offset is not the core problem.
```

### Milestone E: learned retrieval diagnostics

Goal: expose the model’s internal representation and test whether it captures physical similarity.

Tasks:

1. Modify model to return fused embedding.
2. Build synthetic embedding index.
3. Add top-k retrieval to inference.
4. Compare raw pixel nearest-neighbour vs embedding nearest-neighbour vs regression head.
5. Add top-k tolerance metrics.

Gate:

```text
Top-k neighbours should be physically plausible and should usually agree with regression output.
```

### Milestone F: synthetic-realistic domain randomization

Goal: bridge from ideal renderings to real-photo-like synthetic images.

Tasks:

1. Add illumination gradients, vignetting, blur, noise, contrast changes, line thickening, rotation, crop jitter.
2. Train with controlled corruption levels.
3. Test on held-out corruption families.
4. Compare clean-only vs randomized training.

Gate:

```text
Synthetic-realistic performance should degrade gracefully from clean synthetic performance.
```

### Milestone G: real-image preprocessing bake-off

Goal: select robust default preprocessing for real uploads.

Tasks:

1. Build a small real-image suite.
2. Run all preprocessors.
3. Create contact sheets and QA stats.
4. Evaluate prediction stability across preprocessing choices.
5. Select a default CLI pipeline.

Gate:

```text
Real-image outputs should be stable under minor rotations, crops, and threshold changes.
```

### Milestone H: first defect family

Goal: introduce exactly one defect family after the no-defect pipeline is stable.

Recommended first defect: turned edge.

Tasks:

1. Extend manifest schema with optional defect fields.
2. Generate no-defect plus turned-edge synthetic images.
3. Add defect-present and turned-edge parameter heads.
4. Evaluate no-defect regression to ensure it did not regress.
5. Add top-k retrieval including defect metadata.

Gate:

```text
Turned-edge detection should work on synthetic-clean and synthetic-realistic images without degrading no-defect p_corr performance materially.
```

## 16. Practical decision tree

Use this decision tree to avoid chasing the wrong problem.

```text
Start: current model performs poorly on held-out synthetic images

1. Can it overfit 32 synthetic examples?
   No  -> debug labels, loading, preprocessing, optimizer, target scaling
   Yes -> continue

2. Is random synthetic holdout good?
   No  -> increase data quality, model capacity, training stability; test p-only
   Yes -> continue

3. Is structured/grid holdout good?
   No  -> improve grid coverage, conditioning, architecture, interpolation strategy
   Yes -> continue

4. Does oracle offset improve p_corr substantially?
   Yes -> offset ambiguity is central; add offset hints/scans/filtering
   No  -> continue

5. Does embedding retrieval agree with regression?
   No  -> inspect embeddings, losses, architecture, retrieval filters
   Yes -> continue

6. Does synthetic-realistic holdout fail?
   Yes -> add domain randomization and preprocessing robustness
   No  -> continue to real curated images

7. Do real curated images fail?
   Yes -> improve real preprocessing and synthetic-realistic corruption model
   No  -> proceed to first defect family
```

## 18. Future defect strategy, unchanged but deferred

The staged defect plan still makes sense, but it should be deferred until synthetic no-defect performance is strong.

Recommended defect stages:

1. no-defect baseline;
2. turned edge only;
3. single zonal defect;
4. central hill/hole;
5. global correction plus one defect;
6. multiple defects;
7. roughness/astigmatism, if the simulator supports them robustly.

When defects are introduced, use labels like:

```json
{
  "labels": {
    "p_corr": 0.84,
    "defect_present": true,
    "defect_type": "turned_edge",
    "defect_radius": 0.92,
    "defect_width": 0.08,
    "defect_strength": 0.15
  }
}
```

Eventually use model heads:

```text
head_p: p_corr regression
head_off: offset regression
head_defect_present: binary classification
head_defect_type: classification
head_defect_params: radius, width, strength, sign
```

But do not add these until the no-defect pipeline passes synthetic gates.

## 19. Summary of what changed in this revision

The held-out synthetic issue materially changes the project priorities.

### Previous emphasis

```text
The main confounder is likely ideal synthetic images vs messy real images.
```

### Revised emphasis

```text
The real-image gap is still important, but first prove that the current model can solve withheld no-defect synthetic images. If it cannot, the core issue is in data, labels, splitting, target ambiguity, model capacity, or the training objective.
```

### What stays the same

- Do not abandon the CNN approach.
- Do not switch to generic vector search as a replacement.
- Do eventually expose learned embeddings for top-k synthetic retrieval.
- Do keep perfect mirrors as the current phase.
- Do use real-image preprocessing and domain randomization later.
- Do add defects incrementally only after the baseline is stable.

### New immediate next action

```text
Build a clean synthetic diagnostic harness and run the current model unchanged through tiny-overfit, random holdout, structured holdout, p-only, oracle-offset, and nearest-neighbour baselines.
```

That is now the most important next step.
